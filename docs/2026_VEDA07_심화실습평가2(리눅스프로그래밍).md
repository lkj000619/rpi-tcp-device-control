## 1. 실습 개요 및 요구사항
---
### 실습 주제
- TCP 통신을 이용한 원격 장치 제어 프로그램 개발
- 서버 장비: 라즈베리파이 4
- 클라이언트 장비: 우분투 리눅스 PC
### 핵심 구현 요건
- **장치 제어 멀티스레딩**: LED, 부저, 조도센서, 7세그먼트 장치를 제어할 때 입출력 대기로 인해 흐름이 막히지 않도록 독립된 비동기 스레드를 활용해 병렬 처리합니다.
- **공유 동적 라이브러리 연동**: 실질적인 디바이스 제어 명령 수행 라이브러리를 [libRemoteIO.so](file:///home/pang/project_test/lib/) 동적 라이브러리 형식으로 분리 컴파일하여, 하드웨어 기능 업그레이드 시 전체 서버 재빌드 없이 라이브러리 교체만으로 반영되도록 구성합니다.
- **클라이언트 시그널 안전장치**: 실행 도중 예기치 못한 종료 시그널에 의해 연결이 강제 차단되는 것을 막고, 오직 `SIGINT` 입력 시에만 소켓 연결을 안전하게 해제한 후 클라이언트를 릴리즈하도록 보완합니다.
- **서버 데몬 프로세스화**: 원격 제어 서버는 백그라운드에서 실행되는 데몬 프로세스(Daemon)로 구동하며, 터미널 로그를 대신하여 시스템 로그(`syslog`)에 기록을 남깁니다.
- **빌드 자동화 지원**: CMake 및 메이크파일 설정을 도입하여 코드 빌드를 자동화하고, 업데이트된 소스파일만 지능적으로 증분 빌드할 수 있는 스크립트를 구현합니다.
### 장치 제어 목표 기능
- **LED**: 클라이언트 터미널에서 ON/OFF 제어 및 밝기(최대/중간/최저) 조절 제어
- **부저**: 클라이언트 요청에 따라 소리(멜로디 음악) 재생 ON/OFF 제어
- **조도센서**: 조도센서 ADC 값 실시간 모니터링 기능 제공 및 설정한 임계값(Threshold)을 기준으로 어두워지면 보드 D1 LED를 자동으로 점등하고 밝아지면 소등하는 자동 조도 루프 제어
- **7-Segment**: 클라이언트가 전송한 숫자(0~9)를 표현하며, 1초마다 -1씩 차감되는 비동기 카운트다운을 표시하고 카운트가 `0`에 도달하면 알림 부저를 울림

## 4. 개발 진행 로그 및 문제 해결 기록
---
### 74LS47 FND 제어 핀 플로팅 문제
- **문제 현상**: 전선과 빵판을 미세하게 만지면 숫자가 표출되나, 손을 떼면 전체 FND 세그먼트에 불이 켜지며 제어가 불가능해지는 오작동 발생.
- **원인 분석**: 74LS47 BCD 디코더 칩의 내부 특수 핀인 3번(LT, Lamp Test), 4번(BI/RBO, Blanking Input), 5번(RBI, Ripple Blanking Input) 제어 포트를 어디에도 연결하지 않고 공중에 띄워 둠으로써 정전기 노이즈가 유입되어 칩 자체 테스트 모드가 강제 구동됨.
- **해결 방안**: 3번, 4번, 5번 핀을 VCC(5V) 버스로 직접 배선하여 항상 HIGH 상태로 홀딩하는 풀업 조치를 수행하여 안정적으로 BCD 신호만 읽도록 해결함.
### PCF8591 ADC 채널 스캔 지연(Dummy Read) 문제
- **문제 현상**: LDR, NTC, Potentiometer 센서값을 다중 채널로 연이어 요청할 때 센서 간 수치가 혼입되어 다르게 표시되는 버그 발생.
- **원인 분석**: PCF8591 I2C 칩은 내부적으로 채널 레지스터 값을 바꾼 바로 다음 번 읽기 주기(`i2cRead`)에서 변경된 채널의 전압을 리턴하는 것이 아니라, 이전 변환 사이클의 메모리 잔여값을 출력하는 하드웨어 특성이 존재함.
- **해결 방안**: [lib/yl40.c](file:///home/pang/project_test/lib/yl40.c)의 입력값 조회 시, 채널 설정 직후 1차 쓰기/읽기 주기를 무조건 통과시켜 버리고(Dummy Read), 연이은 2차 변환 값을 취함으로써 정확한 수치를 획득하게 조치함.
### 한글 인코딩 및 전송 크기 불안정성 패치
- **문제 현상**: CLI 메뉴 한글이 깨지거나 소켓을 통해 전송받은 텍스트의 끝부분에 깨진 잔여 가비지가 출력됨.
- **원인 분석**: `write(fd, "메시지", N)`에서 바이트 크기 N을 임의의 고정 정수 상수로 명시함으로써, 한글 캐릭터 셋의 3바이트 크기 변화를 동적으로 담지 못해 발생함.
- **해결 방안**: 전송 대상 버퍼 크기를 `strlen("메시지")` 계산식으로 자동 연계하여 가변 바이트 전송을 안전하게 구현함.
### 비동기 스레드 강제 취소 시의 힙 메모리 누수 방지
- **문제 현상**: LED 효과 실행 중 클라이언트가 취소하거나 타이머 중단 시 데몬 서버의 메모리 점유율이 서서히 누적 증가함.
- **원인 분석**: 스레드 생성을 위해 힙 메모리에 할당된 매개변수 구조체를 `pthread_cancel`을 이용해 워커 스레드가 동작 대기(`usleep`) 중인 지점에서 강제로 끊었을 때, 스레드 함수 최종 라인에 걸려 있는 `free(arg)` 블록이 실행 지점 소멸로 인해 도달 불가능해져 누수가 생김.
- **해결 방안**: 각 비동기 워커 스레드가 기동되는 시점(함수 진입 최상단)에서 전달 인자를 스택 지역 변수로 안전하게 복사한 뒤, 대기 및 무한 루프에 돌입하기 바로 전에 `free(arg)`를 선제적으로 완결 실행하여 취소 시에도 누수가 생기지 않는 메모리 안전성을 확립함.
### 실시간 PWM 상태값의 가시성 결여 문제
- **문제 현상**: 클라이언트 제어 메뉴에서 각 LED 및 YL-40 D1 LED의 현재 밝기 상태(PWM 설정값)를 알 수 없어 사용자가 수동으로 매번 밝기 값을 짐작하거나 재입력해야 하는 불편 발생.
- **원인 분석**: 디바이스 제어 데이터 구조체(`readIO_t`)에 일반 LED의 PWM 상태를 저장하는 저장소가 존재하지 않았고, 메뉴 송출 시 실시간 구조체 데이터를 렌더링에 매핑하지 않아 초기값 및 설정값이 공유되지 않음.
- **해결 방안**: [include/common.h](file:///home/pang/project_test/include/common.h) 내 `readIO_t` 구조체에 `led_pwm_state` 배열을 추가하여 3색 LED의 실시간 밝기 상태를 추적하고, LED 동작(ON/OFF/PWM) 시 스레드 안전하게 값을 업데이트하도록 [lib/server.c](file:///home/pang/project_test/lib/server.c) 코드를 보완함. 또한, 상세 제어 메뉴와 밝기 상세 조절 서브 메뉴 진입 시 실시간 PWM 값을 메뉴 스트림에 연동(예: `현재 PWM: %d`)하여 가시성을 확보함.
### 데몬 프로세스의 백그라운드 운용 로그 부재 문제
- **문제 현상**: 백그라운드로 작동하는 서버 데몬 특성상 표준 출력이 모두 차단되어 있어, 클라이언트 접속 실패나 비정상 통신 단선 및 장치 제어 오류 발생 시 원인을 파악하기 매우 어려움.
- **원인 분석**: 데몬 시작 시점에만 syslog를 소량 개방했을 뿐, 실제 소켓 다중 제어 상태를 조율하는 동적 라이브러리 엔진 영역 내에 파일 입출력 로깅 모듈이 존재하지 않았음.
- **해결 방안**: [lib/server.c](file:///home/pang/project_test/lib/server.c) 내에 스레드 안전하게 작동하는 `write_daemon_log` 파일 로그 헬퍼 함수를 추가하고, 프로젝트 루트에 `remoteIO.log` 텍스트 파일을 개방하여 실시간 적재하도록 함. 이를 통해 데몬 시작, 클라이언트 접속(IP 및 소켓 식별값 포함) 및 퇴장/단선, 액추에이터 제어 명령(ON/OFF 및 PWM 밝기값 포함), 조도 오토 모드 피드백 토글 시점 등 주요 트래픽과 하드웨어 변동을 디테일하게 추적할 수 있도록 완성함.
### 타겟 장치(ARM64)와 빌드 장치(x86_64) 아키텍처 불일치 문제
- **문제 현상**: 개발자가 호스트 PC(x86_64)에서 작성한 바이너리를 라즈베리파이 4 타겟 보드에 직접 복제하여 실행할 때, 명령 집합 불일치(Exec format error)가 일어나며 실행에 실패함.
- **원인 분석**: 서버 데몬 프로그램과 디바이스 제어 공유 라이브러리가 타겟 장치의 고유 아키텍처인 ARM64(aarch64) 포맷에 대응하지 못하고 호스트의 x86_64 아키텍처로 컴파일되어 발생함.
- **해결 방안**: 프로젝트 최상위에 CMake 크로스 컴파일용 툴체인 파일인 [toolchain_arm64.cmake](file:///home/pang/project_test/toolchain_arm64.cmake)을 도입하여, 호스트 컴퓨터에 설치된 `aarch64-linux-gnu-gcc` 크로스 컴파일러를 연동하고 타겟 시스템 라이브러리 및 프로그램의 독립 탐색 설정을 격리 구성함. 더불어, [build_all.sh](file:///home/pang/project_test/build_all.sh)를 갱신하여 `./build_all.sh arm64` 파라미터 유입 시 기존 캐시를 자동 정리한 후 툴체인 파일을 활성화해 ARM64 바이너리를 안전하고 편리하게 교차 컴파일할 수 있도록 빌드 인터페이스를 패치함.
## 5. 원격 제어 및 세션 멀티플렉싱 시나리오
---
- **다중 사용자 모니터링**: epoll 멀티플렉싱 설계로 여러 클라이언트가 동시 접속해 조도, 온도, 저항 센서 등의 입력을 간섭 없이 확인 가능합니다.
- **출력 액추에이터 제어 잠금**: 특정 클라이언트가 LED 상세 PWM 제어나 7세그먼트 카운트다운 같은 출력 장치 제어 세션에 진입하면, 서버는 장치 충돌 방지를 위해 점유 락(`active_client_fd`)을 활성화합니다.
- **세션 락 바이패스 및 해제**: 락이 걸린 상태에서 타 클라이언트가 제어를 시도하면 점유 경고와 함께 메인 메뉴로 강제 롤백되며, 기존 사용자가 세션을 상위 메뉴로 빠져나가거나 비정상 연결 종료(소켓 닫힘)되면 즉각 락을 자동 환수하여 타 접속자에게 제어권을 인계합니다.
